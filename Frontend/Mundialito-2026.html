1
Estructura general del archivo

Todo el proyecto vive en un solo archivo .html. Esto es ideal para una maqueta. El archivo tiene tres bloques grandes:

<head>
→ Metadata, fuentes, y todo el CSS dentro de <style>
<body>
→ Todo el HTML visible: header, ticker, tabs, paneles, footer
<script>
→ Los datos y funciones JS que construyen la UI dinámicamente

Cuando el proyecto crezca, vas a separar esto en archivos distintos: index.html, style.css y app.js.

2
Variables CSS con :root

Al inicio del CSS hay un bloque :root con todas las variables de color y tipografía. Esto es lo que define la paleta del Mundial 2026. Si querés cambiar un color en toda la app, lo cambiás en un solo lugar.

/* Así se declaran las variables */ :root { --navy: #0A1628; /* azul marino principal */ --red: #C8102E; /* rojo FIFA */ --gold: #F0A500; /* dorado para rankings */ --white: #FFFFFF; } /* Así se usan en cualquier selector */ .tab-btn.active { color: var(--white); border-bottom-color: var(--red); }

Ventaja enorme: si mañana querés probar con los colores de otro mundial, sólo editás las variables y toda la app cambia instantáneamente.

3
Los datos en JavaScript

Los datos de los partidos, goleadores y tarjetas están guardados como objetos y arreglos de JavaScript. Esto simula lo que en el proyecto real vendría de una API REST.

/* Un grupo es un objeto con dos propiedades */ const GROUPS = { A: { teams: ["Qatar", "Ecuador", ...], matches: [ { home: "Qatar", away: "Ecuador", hs: 0, as: 2, date: "20 Nov" }, // hs = home score, as = away score ] } }; /* Un goleador es un objeto simple */ const SCORERS = [ { name: "Kylian Mbappé", team: "Francia", goals: 8, assists: 2, flag: "🇫🇷" } ];

Cuando conectes la API real, vas a reemplazar estos arreglos con datos que vienen de fetch(). La estructura de la UI no cambia.

4
El sistema de tabs (pestañas)

El sistema de navegación usa una función simple: oculta todos los paneles y muestra solo el seleccionado. Cada botón tiene un atributo onclick que llama a la función con el id del panel.

/* HTML: cada botón pasa su id y referencia a sí mismo */ <button onclick="switchTab('live', this)">En Vivo</button> /* JS: la función gestiona clases CSS para mostrar/ocultar */ function switchTab(id, btn) { // 1. Quita 'active' de todos los paneles y botones document.querySelectorAll('.panel').forEach(p => p.classList.remove('active')); document.querySelectorAll('.tab-btn').forEach(b => b.classList.remove('active')); // 2. Agrega 'active' solo al seleccionado document.getElementById('panel-' + id).classList.add('active'); btn.classList.add('active'); }

En el CSS, .panel tiene display: none y .panel.active tiene display: block. Todo el sistema de navegación es solo eso.

5
Render dinámico con innerHTML

En lugar de escribir el HTML de cada partido a mano, usamos template literals de JavaScript para generarlo desde los datos. La función renderGroups() toma el objeto de datos y construye el HTML como un string.

function renderGroups(key) { const gData = GROUPS[key]; // Genera HTML para cada partido usando .map() const matchesHtml = gData.matches.map(m => ` <div class="gmatch"> <span>${m.date}</span> <span>${m.home}</span> <span>${m.hs} — ${m.as}</span> <span>${m.away}</span> </div> `).join(''); // Lo inyecta en el div correspondiente document.getElementById('grpContent').innerHTML = matchesHtml; }

Este patrón es el corazón de las apps web modernas. Frameworks como React o Vue hacen exactamente esto pero con más optimización. Acá lo hacés a mano para entender el concepto base.

6
El ticker (banda animada)

El ticker de resultados en la parte superior usa solo CSS animation. El truco es duplicar el contenido para que el loop sea infinito y sin saltos.

/* CSS: mueve el contenedor hacia la izquierda indefinidamente */ @keyframes ticker { 0% { transform: translateX(0); } 100% { transform: translateX(-50%); } /* ↑ -50% porque el contenido está duplicado */ } .ticker-track { animation: ticker 28s linear infinite; } /* JS: duplica el HTML para el loop perfecto */ function buildTicker() { const html = items.join(''); track.innerHTML = html + html; // duplicado = loop sin salto }
7
Simulación de tiempo real (minutos)

La app simula los minutos que avanzan en los partidos en vivo usando setInterval(). En el proyecto real, este dato vendría del WebSocket, no de un intervalo local.

/* Cada 5 segundos, incrementa el minuto y actualiza el DOM */ const liveData = [ { minEl: 'live1-min', min: 67, max: 90 }, { minEl: 'live2-min', min: 23, max: 90 }, ]; setInterval(() => { liveData.forEach(d => { if (d.min < d.max) d.min++; document.getElementById(d.minEl).textContent = d.min + "'"; }); }, 5000); // cada 5000ms = 5 segundos /* En producción con WebSocket sería así: */ // socket.on('match-update', (data) => { // document.getElementById('live1-min').textContent = data.minute + "'"; // });
8
Siguiente paso: conectar a una API real

Para pasar de maqueta a app real, hay tres cambios principales:

PASO 1 — API REST
Reemplazar datos hardcodeados
Cambiar los arreglos GROUPS, SCORERS, etc. por llamadas a fetch('https://api-football.com/...') y poblar el DOM con la respuesta.
PASO 2 — WEBSOCKET
Conectar socket.io para tiempo real
Agregar const socket = io('tu-servidor') y escuchar eventos como 'match-update' para actualizar scores y minutos sin recargar la página.
PASO 3 — BACKEND
Servidor Node.js + Express
El backend consulta la API de fútbol cada 30 segundos y emite los datos actualizados a todos los clientes conectados por WebSocket. Actúa como intermediario.
